assistant: [[reply_to_current]] In your setup, the difference is mostly **how OpenClaw authenticates and bills**, not “which company” it talks to.

## Short version

- **`openai/*`** = OpenAI via **API key**
  - billed **pay-as-you-go**
  - uses `OPENAI_API_KEY`
- **`openai-codex/*`** = OpenAI via **Codex/ChatGPT OAuth**
  - billed against your **subscription / Codex entitlement**
  - uses your **logged-in account**, not an API key

## In plain English

### `openai`
This is the normal developer API path.

You give OpenClaw an API key, and it calls OpenAI like a regular paid API client.

That means:
- usage is metered
- cost is per-token / API usage
- good when you want stable API access
- bad if you’d rather burn subscription allowance first

### `openai-codex`
This is the subscription-backed path.

OpenClaw uses your OpenAI/Codex login session via OAuth instead of an API key.

That means:
- it tries to use the **monthly plan / Codex subscription path**
- not direct pay-as-you-go API billing
- depends on OAuth remaining healthy
- can fail if the OAuth token expires or refresh breaks

That’s exactly the path that broke earlier with the `refresh_token_reused` error.

---

## Why both exist

Because they solve different problems:

### `openai-codex` is good for:
- using your existing subscription
- avoiding API spend where possible
- “I already pay monthly, use that first”

### `openai` is good for:
- reliable fallback when OAuth has a tantrum
- direct API usage
- explicit costed access
- automation where API-key auth is simpler

---

## Model naming difference

You’ll often see similar model families exposed under both providers, but they are **different auth/transport lanes**.

For example:

- `openai/gpt-5.4`
- `openai-codex/gpt-5.4`

Same broad model family idea, different provider path.

Think of it like:
- same airline destination
- different ticket counter
- one charges your company account, one charges your personal card
